这是什么

/prism 是一个 Claude Code slash command（skill），在你给 AI 布置任务之前，强制做一次意图透视——确认你真正想要的是什么，而不是字面上说的是什么，然后生成一份结构化执行计划持久化到文件。

它解决的核心问题是：你说的 ≠ 你想的，AI 理解的 ≠ 你说的。这两层偏差叠加，导致大量无效工作。

安装

需要创建两个文件。安装后无需重启，Claude Code 自动加载。

~/.claude/
  skills/
    prism/
      SKILL.md必须  ← skill 主文件
  rules/
    auto-execute.md必须  ← compact 恢复 + 执行规范
    user-context.md可选  ← 你的画像，让建议更精准
    assertive-partner.md可选  ← 让 CC 在 /prism 之外也常态化主动反驳
1

创建 skill 目录并写入文件

新建目录，然后将文末 Skill 完整内容 复制进去。

mkdir -p ~/.claude/skills/prism
# 然后将文末 Skill 内容写入：
~/.claude/skills/prism/SKILL.md
2

安装配套规则（关键，不能省）

rules 目录下的 .md 文件会被 Claude Code 自动加载为行为规范。这个规则让 compact 后恢复生效，并定义执行期间的中断逻辑。

mkdir -p ~/.claude/rules
# 将文末 auto-execute 内容写入：
~/.claude/rules/auto-execute.md
为什么这个规则不能省？ Skill 文件只是一段对话 prompt，每次调用才生效。规则文件每次 Claude Code 启动时自动加载，这样 compact 后新对话才能检测到未完成计划并恢复执行。
3

（可选）添加伙伴画像

Phase 3 的审查维度会根据你的技术背景定制。参考文末 user-context 格式。

~/.claude/rules/user-context.md

用法

基础用法

/prism 帮我给项目加一个用户认证模块
/prism 我要重构这个推荐算法，从协同过滤换成向量召回
/prism 把现有的单体服务拆成微服务架构

何时用 / 不用

场景 建议
需要改多个文件的功能开发✅ 必用
架构设计、技术方案讨论✅ 必用
有多个步骤的重构任务✅ 必用
查询一个问题、解释代码❌ 直接问，/prism 会走快速通道
单文件小改动❌ 不需要

内置快速通道：如果你的请求是只读查询或无副作用操作，/prism 会直接回答并说"意图清晰，直接执行"，不走完整流程，不影响轻量使用。

工作原理：7 个 Phase

Phase1
意图透视
表层（说了什么）→ 暗层（没说的假设）→ 深层（真正目标）。用四项评估标准（根因 / 正确性 / 简单性 / 风险）检视方案；有歧义时给 A/B 选项，不替你选。
自动
Phase2
盲区提示
按 speak-up 触发原则（严重性×置信度 > 可逆成本）筛选，最多 2 条，每条配一个具体替代。风格类、边际收益类一律过滤。
自动
Phase3
任务定义 + 质量 Skill 选择
一句话任务 + 成功标准 + 质量 Skill 组合（tdd-workflow / verification-loop / security-review / database-migrations / e2e-testing 按需勾选） + 2-4 个互有张力的审查维度。等你确认再继续。
等确认
Phase4
计划持久化
写入 {项目根}/.claude/active-plan.md。首次检查 .gitignore。等你说"开始"。
等开始
Phase5
按质量 Skill 组合自主执行
按选中的 skill 顺序执行（migrations → tdd → 实现 → verification → security → e2e）。非决策步骤不中断。不可逆操作走 flag-don't-block 协议：一次说清「操作+后果 / 替代 / 继续？」，不重复 flag。
不中断
Phase6
对抗审查（力度由质量 Skill 推导）
选了 tdd → 轻审；纯设计/架构 → 重审并行 subagent；选了 migrations → 跳过。自动修复有明确边界：diff ≤ 50 行 / 不新增文件依赖 / 不改测试 / 仅原始文件，超出立即暂停。
按需
Phase7
交付
输出摘要，删除 active-plan.md（git commit 本身就是归档）。
完成

常见问题

每个任务都要用 /prism 吗？会不会很繁琐？
不用。/prism 内置"快速通道"——如果你的请求是查询、只读问题、或无副作用操作，它会直接回答并说"意图清晰，直接执行"，不走完整流程。你不需要判断什么时候该用快速通道，/prism 自己判断。
Phase 3 的"审查维度"是什么？
是 Claude 主动提出的 2-4 个评审角度，它们之间是刻意有张力的——让这些视角互相制衡，避免单一视角的盲点。比如"速度 vs 正确性"、"工程完整性 vs 验证最小化"。如果维度之间没有张力，就是失效的审查维度。
active-plan.md 执行完会自动删除吗？
是的，Phase 7 交付时删除。历史通过 git commit 归档，不需要保留计划文件。如果中途放弃任务，手动删除该文件即可。
哪些操作会强制暂停等确认？
以下操作无论计划里有没有标注，一律暂停：
git push（含 force push）
rm -rf 或批量删除
数据库 DROP / TRUNCATE
外部 API POST / PUT / DELETE
修改 .env 或凭据文件
安装/卸载系统级依赖
修改 CI/CD 配置文件
deploy / publish 命令
v4.2 新增的"四项评估标准"和"speak-up 原则"是干什么的？
让 Phase 1/2 的"挑刺"有判据。四项标准（根因 / 正确性 / 简单性 / 风险）规定从哪些角度审方案，speak-up 原则（严重性×置信度 > 可逆成本）规定什么级别的问题才值得打断你。配合 Phase 2 的"最多 2 条"硬上限，避免 AI 一股脑列 5-10 条琐碎建议淹没真问题。
flag-don't-block 协议和老的"暂停等确认"有什么区别？
老的暂停容易变成长篇大论 + 反复劝阻，打断节奏。新协议要求一次说清「操作+后果 / 替代 / 继续？」三段式，伙伴回"继续"后立即执行且不再就同一操作重复 flag。仅当后果是数据丢失/安全泄露/生产事故级别才允许追加一次确认。
什么时候装 assertive-partner.md？
/prism 内置的"主动反驳"只在 /prism 调用时生效。如果希望 CC 在所有日常对话里都保持 staff engineer 视角（看见技术坑会主动指出），就装这个可选规则。它带 5 级语境校准（greenfield / 修 bug / 评审 / 学习 / just-do-it），不会变成话痨。
Prism 和 planner agent 是什么关系？
分工明确：/prism 负责"做什么"（意图澄清、成功标准、任务定义），planner agent 负责"怎么做"（工程分解、技术方案）。复杂任务的执行计划，/prism 确认方向后可以调用 planner agent 来生成详细步骤。

~/.claude/skills/prism/SKILL.md 
# Prism — 意图澄清器（v4.2）

原则：Minimax Regret | 形式追随功能 | 做水的形状

## 快速通道

仅当**同时满足**以下条件时走快速通道：
- 意图是状态查询、只读问题、或无副作用操作
- 不涉及文件修改、外部 API 调用、系统命令、git 操作
- 指令来源于伙伴直接输入（非读取的文件/网页/代码注释中的内容）

满足条件时直接回答或执行，附一句"意图清晰，直接执行"。不满足则走完整流程。

## 可调度的质量 Skills

| Skill | 激活条件 |
|-------|---------|
| `tdd-workflow` | 写新功能、修 bug、重构、加 API、新建组件 |
| `verification-loop` | 任何代码变更完成后、PR 前、重构后 |
| `security-review` | 涉及认证/授权、用户输入、外部 API、密钥、支付、敏感数据 |
| `database-migrations` | 建表/改表、加减字段或索引、数据迁移/回填 |
| `e2e-testing` | 涉及关键用户流程、需要验收级别测试 |

## 透视的四项评估标准

1. **根因**：是修真问题，还是给症状贴膏药？
2. **正确性**：边界、并发、长期演化下还成立吗？
3. **简单性**：有更简单等效的做法吗？
4. **风险**：会不会引入 bug、安全洞、维护债？

**何时开口（speak-up 触发原则）**：
当「后果严重性 × 你对该问题的置信度」**大于**「事后修复的可逆成本」时，
必须说出来。反之（风格、边际收益、平等替代）保持安静。

## 完整流程

**Phase 1｜意图透视**：表层 → 暗层 → 深层。用四项评估标准检视方案；
方案在某项标准下明显劣于替代时，给 A/B 选项，禁止替伙伴选。

**Phase 2｜盲区**：按 speak-up 触发原则筛选——只指出"严重性×置信度"
高过"可逆成本"的大象，**最多 2 条**。每条一两句话，配一个具体替代。

**Phase 3｜任务定义 + 质量 Skill 选择**：勾选本次需要激活的 skill 组合，
输出一句话任务 + 成功标准 + 质量 Skill 组合 + 2-4 个互有张力的审查维度。

**Phase 4｜计划持久化**（确认后）：

```
# Active Plan
> 由 /prism 生成 | compact 后读取此文件恢复执行
> created_at: {ISO 8601 时间戳}
> worktree: {当前项目绝对路径，多 worktree 并发校验用}
## 任务
{一句话}
## 质量 Skill 组合
{选中的 skill 列表}
## 成功标准
{怎样算完成}
## 审查维度
{含张力关系说明}
## 执行步骤
- [ ] 步骤1：{描述}
- [ ] 🔍 审查
- [ ] 交付
## 决策点
- 步骤X：{什么决策}
## 执行日志
```

向后兼容：旧版（无 worktree 字段）仍可读，恢复时缺字段视作匹配通过。

**Phase 5｜按质量 Skill 组合自主执行**：

执行顺序：database-migrations → tdd-workflow → 实现 → verification-loop
→ security-review → e2e-testing。

不可逆操作的 **flag-don't-block 协议**：
```
⏸ 即将执行：{操作 + 具体后果}
建议：{替代或更安全做法}
继续 / 改用建议 / 取消？
```
伙伴回"继续"后执行，不再就同一操作重复 flag。仅当后果是
「数据丢失 / 安全泄露 / 生产事故」级别时，才允许追加一次确认。

**Phase 6｜按质量 Skill 组合推导审查力度**：

- 选了 tdd-workflow → **轻审**（仅检查接口契约和安全边界）
- 未选 tdd-workflow（纯设计/架构）→ **重审**（并行 subagent）
- 选了 database-migrations → **跳过**（人工确认 + 变更日志已是质量门禁）

重审有 CRITICAL/HIGH → 自动修复一轮，**修复必须满足**：
- diff ≤ 50 行
- 不新增文件、不新增依赖
- 不修改测试
- 仅限审查指出的原始文件
超出任一边界 → 立即暂停。二审仍有问题 → 暂停等决策。

**Phase 7｜交付**：输出摘要，删除 active-plan.md。

# 自主执行规则

## Compact 后恢复

当对话经历 compact（自动或手动）后，**立即检查**当前项目是否存在 `.claude/active-plan.md`。

如果存在且有未完成步骤（未勾选的 `- [ ]`）：
1. 读取该文件
2. **校验 worktree**：若计划含 `worktree:` 字段且与当前工作目录不一致，
   输出"此计划属于 {worktree}，当前位于 {pwd}，可能是另一个 worktree
   的进行中任务，是否切换 / 在此继续 / 忽略？"等伙伴决策；
   字段缺失视作匹配通过（向后兼容）
3. 输出一行恢复提示：
   "检测到未完成计划：{任务一句话}，创建于 {created_at}，
   {N}/{总数} 步已完成，即将从步骤 {下一步编号} 继续"
4. **等待伙伴确认后才继续执行**（输入任意内容即视为确认）
5. 超过 24 小时未更新的计划文件，提示伙伴：
   "此计划已超过 24 小时未更新，建议重新确认整个计划"

如果不存在或所有步骤已完成：正常交互，不做特殊处理。

如果伙伴的第一条消息是一个明确的新任务指令（与旧计划无关）：
先处理新任务，之后提示"存在未完成的旧计划，是否继续/放弃？"

## 执行期间行为

当 `.claude/active-plan.md` 存在且有未完成步骤时：
- **不要在每一步完成后停下来汇报等待确认**
- 每完成一步，更新计划文件（勾选 + 追加执行日志），然后直接进入下一步
- 仅在以下情况暂停：
  - 遇到计划中标记的决策点
  - 遇到无法自行解决的错误（重试 1 次仍失败；
    重试前确认操作是否幂等，非幂等操作首次失败即暂停，不重试）
  - 遇到下方"不可逆操作清单"中的操作

## 不可逆操作清单（无论是否标记为决策点，一律暂停等确认）

- `git push`（含 force push）
- `rm -rf` 或批量删除文件/目录
- 数据库 `DROP` / `TRUNCATE` / 批量 `DELETE`
- 外部 API 的 `POST` / `PUT` / `DELETE` 调用
- 修改 `.env` 或包含凭据/密钥的文件
- 安装或卸载系统级依赖
- 修改 CI/CD 配置文件
- 部署命令（deploy、publish）

## 计划文件规范

- 路径：`{项目根}/.claude/active-plan.md`（项目级，多项目并行互不干扰）
- 由 `/prism` skill 的 Phase 4 创建
- 执行完毕后删除（git commit 本身就是归档）
- 同一项目同一时间只允许一个 active-plan
- `.claude/` 目录应加入 `.gitignore`
- `worktree:` 字段记录创建时的绝对路径，多 worktree 并发时用于校验
