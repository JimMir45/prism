/prism for AI builders

让编码 Agent 在动手前，先把问题看清楚

这不是另一个“更会写代码”的 skill，而是一个给 AI 应用开发流程加上的 preflight layer：先校准意图，再生成计划，再执行。

面向对象：AI 应用开发团队
载体：Claude Code skill
核心承诺：减少跑偏和返工

这场分享回答三个问题

  • 意图：它到底想纠正 AI coding workflow 的什么问题？
  • 作用：它如何改变任务定义、执行和交付？
  • 边界：它什么时候很值，什么时候反而拖慢？
7 个 phase，把“模糊想法”拉直到“可恢复执行”
1 个核心文件：`.claude/active-plan.md`
项目定位：意图澄清器 + 计划持久化器 + 风险闸门
建议演讲时长：15-20 分钟
Problem Framing
02 / 10

AI 应用开发里，真正昂贵的不是写代码，而是 写错代码

团队做 AI 产品时，经常不是“不会实现”，而是过早实现了一个错误假设：用户场景不对、权限模型没定义、评估标准没对齐、调用外部资源的风险没说清。

需求语义太短
方案空间太大
误操作代价高
上下文经常丢

它瞄准的四个常见失败模式

  • 字面执行：Agent 按表述写，不按真实目标做。
  • 方案抢跑：还没澄清，就已经选了技术路线和数据模型。
  • 长任务失忆：会话 compact 后，执行上下文断裂。
  • 自主过头：危险操作发生前，没有一个明确暂停点。

一句话理解：/prism 不是提升单次生成质量，而是减少整个任务生命周期中的“方向性错误”。

适合 AI builder 的原因：我们经常处理 高不确定性 + 高耦合 的任务。
问题不是 token 不够，是前置判断不够。
Definition
03 / 10

/prism 是什么

一个给 Claude Code 用的 slash command skill，在执行前强制做一次 意图透视，并把后续计划写入可恢复文件。

/prism 我要做一个用户管理后台

目标不是让模型“更像人”，而是让它在面对模糊任务时，先像一个谨慎的 staff engineer。

它不是什么

  • 不是通用 agent 框架，也不替代现有 coding workflow。
  • 不是知识库或 RAG，重点不在补信息，而在补 决策前置。
  • 不是为了所有请求都更慢更严谨，而是对 模糊且代价高 的任务加一道闸门。

核心输出

  • 结构化任务定义：一句话任务、成功标准、审查维度。
  • 执行计划：写到 `.claude/active-plan.md`，支持跨 session 恢复。
  • 暂停协议：危险操作前强制确认。
本质：把“prompt”升级成一个 任务操作协议。
对团队价值：可复用、可解释、可恢复。
Mechanism
04 / 10

7 个 Phase，把模糊任务转成可控执行

  • Phase 1 意图透视：表层、暗层、深层，必要时给 A/B 选项。
  • Phase 2 盲区提示：只提高严重度且高置信度的问题。
  • Phase 3 任务定义：明确成功标准、审查维度、要激活的质量 skill。
  • Phase 4 计划持久化：写入 `active-plan.md`。
  • Phase 5 自主执行：非决策步骤不停顿。
  • Phase 6 对抗审查：根据任务类型决定轻审/重审。
  • Phase 7 交付：总结并清理计划文件。

前置对齐

先判断“用户真正要什么”，而不是顺着第一个措辞直接开工。

计划留痕

把中间判断沉淀成文件，而不是只存在上下文窗口里。

自主但有限

允许连续执行，但把不可逆动作集中到可确认的停点。

轻重审查可调

有测试的任务轻审，没有测试或偏架构的任务重审。

兼容技能编排

可根据任务激活 TDD、verification、security-review 等子技能。

有快速通道

纯查询或无副作用任务可直接执行，避免一刀切过度流程化。

关键设计点：不是每次都慢，而是在该慢的时候慢下来。
这和 agent autonomy 并不冲突。
Before / After
05 / 10

同一句话，不同的处理方式，会导向完全不同的工程成本

Vanilla Claude Code

先做，再发现做偏

“做一个用户管理后台” 好的，我来用 React + Express 搭一个后台， 先建 users 表、角色表、登录页和 CRUD 页面……
  • 默认把“后台”理解成一个完整新系统。
  • 默认把“管理”理解成 CRUD，不问审计和权限矩阵。
  • 技术选型已经落地，后续纠偏成本陡增。
With /prism

先定义问题，再决定怎么做

Phase 1 透视到三种可能： A. 内部运维后台 B. 多租户 SaaS 管理台 C. 现有 App 的 admin 入口 Phase 2 盲区： “管理”是否包含审批、审计、权限矩阵？ 请先选择 A/B/C 并说明权限模型。
  • 先缩小问题空间，避免提前绑定方案。
  • 把真正影响架构的变量拉到台前。
  • 只有当目标清晰后，执行才有意义。
对 AI app 团队最有价值的点：它处理的是 ambiguity，不只是 syntax。
省下的是返工时间，不只是 prompt token。
Impact
06 / 10

1. 把“需求沟通”嵌进命令行

它让 slash command 不再只是一个触发器，而是一个要求先完成任务定义的协议入口。

2. 把模糊成本显性化

通过 A/B 选项、盲区提醒和审查维度，让隐性分歧尽量在动手前暴露。

3. 把长任务从上下文里“落盘”

`active-plan.md` 让 compact 不再等于失忆，特别适合长链路实现和多人接力。

4. 把 agent autonomy 收束到安全范围

允许连续推进，但在危险动作前统一使用暂停协议，而不是随机发挥。

5. 把质量要求前置编排

把 TDD、verification、security-review 等质量动作纳入任务定义，而非收尾补救。

6. 让“像资深工程师”可复制

它把 staff engineer 的判断框架写成了可执行 prompt，不依赖单个使用者的经验密度。

关键词：对齐、留痕、护栏。
适合作为 AI coding 的上层控制面。
Strengths
07 / 10

这个 skill 的主要优点

  • 减少大方向跑偏：在方案空间很大时，先问关键分叉。
  • 风险意识强：不可逆操作被明确收束，而不是靠使用者记得提醒。
  • 长任务稳定：计划文件让任务能恢复、追踪、切换 worktree 校验。
  • 有工程纪律：成功标准、质量 skill、审查维度都在执行前定义。
  • 适合团队复制：比“大家多想想”更可操作，因为它已经把思路模板化了。

对 AI 应用开发团队尤其有用的场景

  • 产品探索期：一句话需求背后有多条可行路线。
  • 代理系统改造：调度、记忆、工具边界常常没有一次说透。
  • 数据和权限相关任务：一旦误解，返工成本高。
  • 需要多人串接的长任务：计划落盘后更容易协作和交接。
  • 希望提升 agent 可靠性：不是只追求快，而是追求少犯高成本错误。

本质收益：它用少量前置摩擦，换来更低的后置返工和更少的隐性风险。

优点不是“更万能”，而是 更不容易在重要任务上做错第一步。
这对 AI 产品研发很关键。
Trade-offs
08 / 10

代价是真实存在的，不能把它讲成银弹

  • 启动更慢：意图已足够清晰时，完整流程会增加额外回合。
  • 依赖 prompt 纪律：它不是编译器，行为稳定性仍受模型和上下文影响。
  • 生态绑定明显：当前是为 Claude Code 的 skill / rule 机制设计的。
  • 有时会“过度谨慎”：如果团队偏好快试快错，可能觉得它挡手。
  • 维护成本不为零：SKILL、rules、evals 都需要持续演进和回归验证。

需要诚实强调的限制

不能替代真实产品判断
不能保证每次都问到关键问题
不能自动变成团队共识

如果组织本身没有意图澄清习惯，只装一个 skill，不会神奇地产生工程判断力。它更像一个放大器：把好的工程习惯制度化，也把坏的输入暴露得更早。

所以它最适合的组织前提

  • 愿意为高价值任务多花 1-2 个回合澄清。
  • 认可“先定义成功标准，再执行”的协作方式。
  • 把 agent 当工程协作者，而不是纯代码生成器。
讲优点时一定同步讲代价，否则团队预期会失真。
它优化的是 可靠性，不是极限吞吐。

Takeaways
10 / 10

结论

/prism 的真正价值，不是让模型“更聪明”，而是让团队在使用 coding agent 时，拥有一个更像工程系统的 前置决策层。

  • 它解决的是 任务定义不清、长任务失忆、危险动作无闸门。
  • 它最适合 AI 应用开发中那些模糊、昂贵、耦合高的任务。
  • 它的缺点也明确：增加前置摩擦，不适合一切场景。

如果团队要试用，我会这样落地

  • 先小范围试点：从数据、权限、长链路任务开始。
  • 保留快速通道：小修小补不要强行走全流程。
  • 配合 evals：改 `SKILL.md` 后跑行为回归，防止风格漂移。
  • 把它当 protocol，不只是 prompt：重点是组织协作方式的改变。

最后一句：对于 AI 编程，真正稀缺的不是“更快写出代码”，而是“更早知道不该写什么代码”。
